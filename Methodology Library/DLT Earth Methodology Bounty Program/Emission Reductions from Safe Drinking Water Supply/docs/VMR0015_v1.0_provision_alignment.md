# VMR0015 v1.0 — Provision Alignment & Test-Data Justification

**Policy:** VMR0015 v1.0 Safe Drinking Water dMRV (Hedera Guardian)
**Methodology:** Verra VMR0015 *Revision to AMS-III.AV.: Low Greenhouse Gas Emitting Safe Drinking Water Production Systems*, **v1.0, active since 31 October 2025** (replaces standalone AMS-III.AV.; AMS-III.AV. v9.0 registrations must validate by 1 May 2026).
**Test project:** VCS 3599 — *Grouped Projects for Safe Drinking Water for Schools in Viet Nam* (real registered Verra project; linked to the World Bank Emission Reduction-Linked Bond for clean water in Vietnam).
**Monitoring period modelled:** 01 Jan – 30 Jun 2025.

This document maps each VMR0015 v1.0-specific provision (i.e. what v1.0 added on top of the underlying AMS-III.AV. equations) to how this Guardian policy handles it, or — where a provision is not exercised by the VCS 3599 test data — states the scope exclusion explicitly so reviewers can see the decision rather than infer it.

## 2. VMR0015 v1.0-specific provisions

### 2.1 Leakage adjustment factor set at validation
**v1.0 requirement:** the leakage adjustment factor accounting for non-renewable woody biomass saved by the project must be **set at validation**.
**Handling:** Leakage enters the on-chain calculation through the Monitoring Report **`field5` (LE_y, Leakage Emissions total, tCO₂e)**, deducted in Eq.7. For the VCS 3599 test, LE = 8,116.00 tCO₂e (validation-set value carried into the monitoring report). The factor itself is a validation-stage determination; the policy consumes the resulting LE figure rather than recomputing it on-chain, consistent with the methodology placing the factor at validation.
**Status:** Implemented (as the LE input); validation-stage origin documented.

### 2.2 Updated emission factor for non-renewable woody biomass / fNRB (TOOL33)
**v1.0 / CDM context:** the AMS-III.AV. v9.0 revision **removed the reference to TOOL30 and added TOOL33**, which supplies default fNRB values; VMR0015 v1.0 updated the emission factor for non-renewable woody biomass (CO₂ and non-CO₂).
**Handling:** `field16` (EF_fuel, tCO₂/TJ) and `field17` (f_i = fNRB) are explicit monitored inputs. The test uses fNRB = 0.82 and EF = 81.6 tCO₂/TJ. fNRB is **TOOL33-aligned** (default-value sourced, not TOOL30). The policy clamps fNRB to ≤ 1.0 and treats fNRB ≤ 0 as zero fossil-displacement credit (conservative).
**Status:** Implemented as explicit inputs; TOOL33 alignment noted. *Action for PR text: state the TOOL33 fNRB source/value used.*

### 2.3 Double-counting assessment with REDD+ / jurisdictional REDD+
**v1.0 requirement:** assess double counting of emission reductions with REDD+ projects and jurisdictional REDD+ programs.
**Handling (scope exclusion):** VCS 3599 is a **point-of-use water-purification** project (household/school water treatment displacing fuel for boiling). It does **not** generate forestry/REDD+ credits and does not overlap a REDD+ or jurisdictional REDD+ crediting boundary, so there is **no double-counting pathway to assess** for this test project. For a forestry-adjacent project, this assessment would be a validation-stage attestation captured in the Project Description.
**Status:** Not applicable to VCS 3599 — documented scope exclusion.

### 2.4 Baseline adjustment for interacting technologies
**v1.0 requirement:** adjust the baseline level of the residence/institution to account for effects of interacting technologies.
**Handling:** Captured by **`field14` (X_boil)** — the fraction of the served population whose baseline practice is/would be water boiling. For VCS 3599 (Case 1/2 institutional water purifiers in schools where boiling is the baseline) X_boil = 1.0; where interacting technologies reduce the boiling baseline, X_boil < 1.0 proportionally reduces creditable BE. No competing low-emission treatment technology is present in the VCS 3599 baseline, so X_boil = 1.0 is appropriate.
**Status:** Implemented (via X_boil); test value justified.

### 2.5 Compilation/presentation of relevant data per project
**v1.0 requirement:** present relevant monitored data per project instance.
**Handling:** The Monitoring Report schema captures all monitored parameters (`field10`–`field18`) per reporting period as a signed Verifiable Credential; the full lifecycle (PP → VVB → SR → mint) produces a Verifiable Presentation and Trustchain anchoring every value to its signer.
**Status:** Implemented.

## 3. VMR0015 §6.1 water-quality eligibility gate

**Requirement:** ERs cannot be claimed for any monitoring period in which >10% of sampled appliances fail water-quality requirements.
**Handling:** `appliance_pass_rate = field10 / field11`; if pass-rate < 0.90, ER is forced to 0 (**fail-closed**, including on missing/zero sampling data).
**Validation:** `policy_integrity_test.js` tests T2 (85% → 0), T3 (90% boundary → credits), T4 (no sampling → 0).
